# EchoVerse 安全说明

## 支持范围

EchoVerse Launcher 当前处于本地原型阶段，尚未发布受支持的生产版本。原型只允许用于受控开发和演示，不应承载真实支付、未成年人敏感数据、生产游戏权限或高价值账户。

| 版本 | 安全支持 |
| --- | --- |
| `0.x` 原型 | 尽力修复；不提供生产安全保证 |
| 未发布的工程化分支 | 仅供开发验证 |

## 报告安全问题

请通过仓库所有者指定的私密渠道报告漏洞，不要在公开 Issue、讨论区、截图或演示视频中披露可利用细节、个人资料或密钥。报告应尽量包含：

- 受影响的提交或版本；
- 复现条件和最小复现步骤；
- 可能影响的数据或能力；
- 已知缓解措施；
- 联系方式和期望披露时间。

仓库所有者应确认收到报告、评估严重度、准备修复与回归测试，并在修复可用后协调披露。安全问题未解决前不得将原型描述为生产可用。

## 密钥和凭据

- 禁止向 Git 提交 API Key、密码、访问令牌、私钥、账户数据库或真实会话文件。
- `config/*.local.json` 只用于本地开发，并必须保持在 `.gitignore` 中。
- 当前开发用模型配置应放在 Electron `userData/ai.json`，不得打入安装包或备份包。
- 环境变量、终端历史、聊天、截图、日志和构建产物同样可能泄密；任何曾暴露在这些位置的密钥都应立即撤销并轮换。
- 示例配置只能包含明确的占位符，测试使用不可访问真实服务的假值。
- 正式版本不得向客户端分发模型供应商密钥；密钥必须由平台 AI 网关保管。
- 发布前应扫描 Git 历史、工作树、`dist`、`dist-electron`、`release` 和压缩备份。

如果发现密钥泄露：先撤销/轮换，再清理分发渠道和历史，最后排查调用记录与费用。仅从最新提交删除字符串并不能使旧密钥恢复安全。

## 当前安全模型

### 已有的本地基线

- 密码保存为随机盐和 `scrypt` 哈希，不保存明文。
- 登录失败有单进程内的临时频率限制。
- 账户写操作在单实例中串行化，并使用临时文件替换和 `.bak` 回退。
- Electron 使用 `contextIsolation`、沙箱和关闭 Node 集成的渲染窗口。
- 主进程限制新窗口、外部导航、WebView 和权限请求，并校验 IPC 发送窗口与页面来源。
- 渲染层只能通过 preload 暴露的窄接口调用主进程。
- AI 请求有输入长度、历史条数、输出长度和超时限制。
- 游戏会话文件权限尽量收紧、有效期短，并在失败/退出/过期后清理。

这些措施只降低误用和常见攻击面，不会把本地原型变成安全的联网账户系统。

### 当前明确不覆盖的威胁

- 攻击者拥有当前 Windows 用户权限、调试权限或磁盘访问权。
- 多进程、多设备或网络环境下的账户一致性和会话撤销。
- 客户端二进制被篡改、注入、重打包或绕过本地规则。
- 本地 `prototype-local` ticket 被伪造。
- 模型供应商对所发送资料的存储、训练或跨境处理合规性。
- DDoS、撞库、凭据填充、支付欺诈和大规模滥用。

## 敏感数据处理

账户邮箱、现实资料、AI Agent 档案、聊天内容、密码哈希和游戏 ticket 均按敏感数据处理。

- 只收集完成当前功能所需的字段；空白字段保持未知，不推断为事实。
- UI 中“不可修改”是产品规则，不是保留数据的合法依据。正式版仍需提供适用法律要求的数据访问、更正、导出和删除流程。
- 日志使用账户/请求的不可逆关联标识，不记录邮箱、完整提示、聊天正文、密钥或 ticket。
- 错误消息不得回显绝对路径中的用户名、供应商响应体或配置内容。
- 备份和崩溃转储必须遵守与主数据相同的访问和删除策略。
- 将玩家资料发送给模型前，正式版必须取得清晰同意，并披露发送字段、目的、供应商和保留期限。

## Electron 变更要求

新增窗口或 IPC 通道时必须：

1. 默认使用 `sandbox: true`、`contextIsolation: true`、`nodeIntegration: false`。
2. 只在 preload 中暴露具体业务方法，禁止暴露通用 IPC 或 Node API。
3. 在主进程验证调用窗口、顶层 Frame、Origin/本地文件路径和输入结构。
4. 对导航、新窗口、WebView、下载和权限采用默认拒绝策略。
5. 为未登录、错误窗口、错误 Origin、超长输入和并发调用添加负向测试。
6. 配置严格 CSP；生产界面不得依赖在线字体或任意远程脚本。
7. 审查返回值，确保不把盐、哈希、密钥、内部路径或供应商原始错误传给页面。

## AI 服务变更要求

- 供应商 Base URL 必须使用 HTTPS 且在明确白名单内，开发用本机 mock 需单独开关。
- provider、API 路径与模型必须兼容，不能把某供应商模型名发送到另一供应商。
- 对每账户和全局请求设置并发、速率、超时和费用限制。
- 人格资料作为数据序列化，提示中明确其内容不是可执行指令。
- 只发送必要字段和有限历史；默认不启用供应商存储或训练。
- 记录请求 ID、耗时、模型和结果类别，不记录提示正文或密钥。
- 对提示注入、越权索取资料、供应商超时、429、5xx、空响应和取消操作建立测试。

## 游戏启动变更要求

- UE5 不得接收账户密码、刷新令牌或模型密钥。
- `prototype-local` 票据只允许 Development 构建；Shipping 构建必须拒绝。
- 正式 ticket 必须短时、一次性、绑定账户/客户端/用途，并通过 HTTPS 后端交换。
- 会话文件路径视为敏感参数，不记录到公共日志，不允许从不可信页面自由指定。
- 可执行路径和附加参数来自受控配置；新增参数必须审查命令行注入和路径边界。

## 依赖与发布

- 使用锁文件和固定工具链；CI 必须执行冻结安装。
- 每次发布执行 `pnpm audit --prod --audit-level high`，高危和严重漏洞必须修复或形成书面例外。
- 依赖升级应审查变更日志、许可证、安装脚本和供应链风险。
- 发布包必须来自干净提交，由 CI 或可审计构建机生成，并记录 SHA-256。
- 生产发布需要 Windows 代码签名、可信下载渠道、升级签名验证和回滚方案。
- 安装包不得包含源码开发配置、测试账户、`accounts.json`、`ai.json`、会话文件或历史 `dist` 残留。

## 安全发布阻断条件

出现以下任一情况不得发布生产版本：

- 客户端或产物包含真实供应商密钥；
- 仍使用本地 JSON 作为生产账户认证来源；
- UE5 Shipping 构建接受 `prototype-local` ticket；
- 无法说明发送给 AI 的个人数据及删除方式；
- 高危/严重生产依赖漏洞未处理；
- Electron 重新开启 Node 集成、关闭隔离/沙箱，或敏感 IPC 缺少来源校验；
- 产物未签名、无法关联到唯一提交，或没有可执行的回滚方案。
