# EchoVerse Launcher 工程审计

审计日期：2026-07-17  
审计对象：Windows Electron 启动器原型  
恢复基线：`v0.2.0-prototype-ai-avatar`  
工程化分支：`codex/engineering-hardening`

## 结论

当前版本已经能够演示注册、登录、首次资料、AI 人格对话、3D 虚拟分身、伴生模式和 UE5 启动交接，但它仍是单机原型，不具备互联网生产账户平台所需的认证、隐私治理、服务端密钥保护、代码签名和运维能力。

本轮工程化工作的目标是建立“可重复安装、可自动检查、可回滚”的开发基线，并收紧本地原型的明显风险。它不会把本地 JSON 账户库或客户端直连模型升级成生产服务。

## 主要风险与处理优先级

| 优先级 | 风险 | 当前原型 | 生产门槛 |
| --- | --- | --- | --- |
| P0 | 客户端持有模型供应商密钥 | 开发机从用户数据目录读取本地配置并直连模型 | 密钥仅存后端；启动器调用受认证、限流、审计的 AI 网关 |
| P0 | 本地账户不是真实平台认证 | 账户保存在单机 JSON；会话只存在于当前进程 | 服务端账户、邮箱验证、密码重置、会话撤销、MFA/风控和审计日志 |
| P0 | UE5 票据不具备认证能力 | `prototype-local` 票据只用于联调 | 服务端签发短时、一次性 ticket；Shipping 构建必须在线交换并拒绝原型票据 |
| P0 | 个人资料与 AI 数据治理不足 | 已默认排除真实姓名、精确生日、居住地和信仰，但仍会向供应商发送有限人格信号与聊天内容 | 明示同意、可配置字段最小化、隐私政策、导出/删除、保留期限和供应商数据处理协议 |
| P1 | Windows 发布可信度 | portable 包尚无签名与自动更新体系 | 代码签名、可信下载、升级/回滚策略、恶意软件误报处理 |
| P1 | 缺少端到端验证 | 当前以模块测试和人工测试为主 | 至少覆盖注册→资料→主页→退出、AI 失败、伴生模式和游戏启动的 E2E |
| P1 | 故障可观测性仍不完整 | 已有本地主进程事件日志，尚无用户诊断导出、崩溃上报或指标 | 脱敏日志导出、崩溃报告、请求关联 ID、指标告警和支持流程 |
| P1 | UI 工程边界仍需继续收敛 | 页面和组件已拆分，但全局样式仍较大，页面状态切换会卸载 | 按领域拆分样式和状态；建立组件、E2E 与可访问性规范 |
| P2 | 3D 资源和渲染成本 | 单个临时角色适合演示 | 资产版本、压缩预算、低端机档位、帧率/显存基线和加载失败降级 |
| P2 | 数据模型演进 | 账户库版本固定为 `1` | 可回滚迁移、备份恢复测试和向前/向后兼容策略 |

## 当前原型可以承诺什么

- 支持单机、单实例的开发演示。
- 账户密码不会以明文写入账户文件。
- 首次保存的非空个人资料不可在当前 UI/API 中修改，空白项可后续补充。
- 新账户获得独立的 Agent ID、人格档案和模型分配元数据。
- Launcher 可以通过版本化会话文件向开发版 UE5 交付账户与 Agent 信息。

## 当前原型不能承诺什么

- 不能抵御拥有本机账户或文件系统权限的攻击者。
- 不能作为联网游戏的身份认证、支付、未成年人保护或封禁依据。
- 每个账户拥有独立 Agent 档案、记忆命名空间和人格上下文，但所有账户可共享同一基础模型；这不代表每个玩家部署了一套独占模型权重或算力实例。
- 本地 `ai.json` 仅用于开发，不是安全的客户端密钥分发方案。
- 本地会话文件中的原型 ticket 不能证明玩家身份。
- 尚未形成正式 SLA、灾难恢复目标或隐私合规结论。

## 合并前验证清单

1. 从无 `node_modules` 的干净副本执行 `pnpm install --frozen-lockfile`。
2. 执行 `pnpm check`，所有类型检查、测试和生产构建必须通过。
3. 执行 `pnpm audit --prod --audit-level high`，高危及严重漏洞为零。
4. 执行 `pnpm dist:win`，在另一台 Windows 机器验证 portable 包。
5. 扫描仓库、构建目录和发布包，确认没有密钥、账户库、聊天记录或本机绝对路径。
6. 人工验证账户资料不可覆盖、AI 未配置/超时/错误响应、3D 加载失败、伴生模式恢复和 UE5 不存在时的错误提示。
7. 记录产物名称、Git 提交、SHA-256、测试机器和已知问题。

## 下一阶段建议

下一阶段应优先设计平台后端边界，而不是继续把生产职责加入 Electron 主进程。推荐先定义账户 API、AI 网关 API、一次性游戏票据交换和隐私数据生命周期，再分别替换本地账户库、客户端模型密钥和原型 ticket。界面拆分、E2E、签名发布和可观测性可在这些接口稳定后并行推进。
